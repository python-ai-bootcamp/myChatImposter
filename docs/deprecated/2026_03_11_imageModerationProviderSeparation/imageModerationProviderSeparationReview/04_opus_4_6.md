## Detailed Findings

### <a id="find-001"></a>FIND-001: `ModerationResult` type used but never defined

**Priority**: 🔴 P1  
**ID**: FIND-001  
**Status**: pending

**Description**:  
`ModerationResult` is the return type of `ImageModerationProvider.moderate_image()` (Section 3.3) and the example consumer code (Section 4.2). It is never defined — no class, no import, no stub.

The OpenAI SDK returns `openai.types.ModerationCreateResponse`. The spec needs to either define a custom wrapper or declare it as a pass-through of the SDK type.

Without this, nobody can implement `OpenAiModerationProvider` or write consumer code.

**Required Action**:  
- Define `ModerationResult` (Pydantic model) and specify its fields.

---

### <a id="find-002"></a>FIND-002: Factory dynamic module resolution will load the wrong provider for `image_moderation` tier

**Priority**: 🔴 P1  
**ID**: FIND-002  
**Status**: pending

**Description**:  
The factory uses `importlib.import_module(f"llm_providers.{config.provider_name}")` to load providers. After the refactor this becomes `model_providers.{provider_name}`.

The problem: `image_moderation` configs will likely have `provider_name: "openAi"` (same as chat configs). The factory would load `model_providers/openAi.py`, which contains `OpenAiChatProvider` — **not** `OpenAiModerationProvider` (which lives in `model_providers/openAiModeration.py`).

The factory has no tier-aware dispatch. It will instantiate a chat provider when a moderation provider is needed.

**Required Action**:  
- require `provider_name: "openAiModeration"` in image_moderation configs (matching the module filename), and update migration script to set this value.

### <a id="find-003"></a>FIND-003: `OpenAiModerationProvider` API key resolution logic not specified

**Priority**: 🟠 P2  
**ID**: FIND-003  
**Status**: pending

**Description**:  
Section 3.5 says `OpenAiModerationProvider` uses `AsyncOpenAI` directly, but never specifies how the API key is resolved. The existing `OpenAiLlmProvider._build_llm_params()` has non-trivial logic for `api_key_source` ("environment" vs "explicit") with validation and fallback.

`OpenAiModerationProvider` needs the same key resolution but has no access to the chat provider's helper method. The spec should specify whether this logic is duplicated, shared via a utility, or inherited from the base class.

**Required Action**:  
- specify that `OpenAiModerationProvider` resolves the API key similarly to existing LLM providers.

### <a id="find-004"></a>FIND-004: `ChatCompletionProviderSettings` silent on `Config.extra = 'allow'`

**Priority**: 🟠 P2  
**ID**: FIND-004  
**Status**: pending

**Description**:  
The existing `LLMProviderSettings` has `Config.extra = 'allow'`, which prevents Pydantic `ValidationError` when database documents contain unexpected fields.

The spec explicitly states `BaseModelProviderSettings` drops `extra = 'allow'` (justified by the migration script). But the spec is **silent** on `ChatCompletionProviderSettings`. If it also drops `extra = 'allow'`, any `high`/`low` tier database documents with stray fields will break deserialization.

**Required Action**:  
- Explicitly state that `ChatCompletionProviderSettings` drops `extra = 'allow'`

---

### <a id="find-005"></a>FIND-005: `ChatCompletionProviderConfig` doesn't inherit from `BaseModelProviderConfig`

**Priority**: 🟠 P2  
**ID**: FIND-005  
**Status**: pending

**Description**:  
Both config classes define `provider_name: str` independently — `ChatCompletionProviderConfig` does not inherit from `BaseModelProviderConfig`. This means they are **unrelated types** in Pydantic's type system.

The factory needs `provider_name` from whichever config it receives. Without a common base type, every piece of code that handles "any provider config" must use `Union[ChatCompletionProviderConfig, BaseModelProviderConfig]` and duplicate attribute access patterns.

If `ChatCompletionProviderConfig` inherited from `BaseModelProviderConfig` (overriding `provider_config` to `ChatCompletionProviderSettings`), the factory could uniformly accept `BaseModelProviderConfig` and Pydantic would handle the field override correctly since `ChatCompletionProviderSettings` extends `BaseModelProviderSettings`.

**Required Action**:  
- Either make `ChatCompletionProviderConfig` inherit from `BaseModelProviderConfig`, or document why they are intentionally separate and how the factory handles the dispatch.

### <a id="find-006"></a>FIND-006: `extractor.py` / `runner.py` parameter cascade not addressed in spec

**Priority**: 🟠 P2  
**ID**: FIND-006  
**Status**: pending

**Description**:  
The spec details the `AutomaticBotReplyService` async refactor (Section 4.5.1) but doesn't address the **more complex** `extractor.py` refactor. The extractor makes **two** factory calls (low and high tier) and its `extract()` method currently accepts 5 parameters that all become dead after the factory redesign:

```python
async def extract(self, messages, llm_config: LLMProviderConfig, user_id: str, bot_id: str,
                  ..., llm_config_high: LLMProviderConfig = None,
                  token_consumption_collection = None)
```

After the factory change, both LLM calls collapse to:
```python
llm = await create_model_provider(bot_id, "periodic_group_tracking", "low")
high_llm = await create_model_provider(bot_id, "periodic_group_tracking", "high")
```

This cascades upward — `runner.py` (lines 171-186) currently resolves `llm_config`, `llm_config_high`, `owner_user_id`, and `token_consumption_collection` before passing them to `extractor.extract()`. All of that resolution code becomes dead. Similarly, `GroupTrackingRunner.__init__` and `GroupTracker.__init__` both accept `token_consumption_collection` which is no longer needed.

**Required Action**:  
- Add a section (parallel to 4.5.1) documenting the `extractor.extract()` parameter simplification and the `runner.py` → `GroupTrackingRunner` → `GroupTracker` cascade of dead `token_consumption_collection` parameters.
