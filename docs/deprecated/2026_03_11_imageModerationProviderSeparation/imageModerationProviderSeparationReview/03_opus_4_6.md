### <a id="gap-001"></a>GAP-001: `record_model_interactions` nomenclature mismatch with codebase

**Priority**: 🔴 P1  
**ID**: GAP-001  
**Status**: pending

**Description**:  
The spec (Section 2.2) introduces `ChatCompletionProviderSettings` with a field named `record_model_interactions`. However, the **current codebase** uses `record_llm_interactions` everywhere:

- `config_models.py` line 53: `record_llm_interactions: bool`
- `llm_providers/openAi.py` line 16: `llm_params.pop("record_llm_interactions", None)`
- `features/periodic_group_tracking/extractor.py` line 104: `llm_config.provider_config.record_llm_interactions`
- `tests/test_e2e.py` lines 69, 78
- `routers/bot_management.py` line 464
- `scripts/migrate_image_moderation.py` line 24

The spec never explicitly calls out this rename from `record_llm_interactions` → `record_model_interactions`, yet the code snippet in Section 2.2 uses the new name. This will cause a **silent field breakage** across the codebase and stored database documents unless:
1. The rename is **explicitly listed as a breaking change** in Section 4.4 or 4.5.
2. A database migration script is specified to update existing documents.
3. All downstream references are cataloged.

**Required Action**: 
- Modify spec accordingly to leave the existing `record_llm_interactions` name as is without changing it at all (since it only applies to chat completion providers, the "LLM" semantics are still accurate)

### <a id="gap-002"></a>GAP-002: `QuotaService.calculate_cost` only accepts `Literal["high", "low"]` — not just `TokenConsumptionService`

**ID**: GAP-002

def calculate_cost(self,

### <a id="gap-003"></a>GAP-003: `BaseLlmProvider.__init__` signature drops `user_id` silently — breaks `OpenAiLlmProvider` constructor chain

**Priority**: 🔴 P1  
**ID**: GAP-003  
**Status**: pending

**Description**:  
The current `BaseLlmProvider.__init__` (in `llm_providers/base.py` line 11) accepts two parameters:
```python
def __init__(self, config: LLMProviderConfig, user_id: str):
    self.config = config
    self.user_id = user_id
```

The spec's new `BaseModelProvider.__init__` (Section 3.1) only accepts `config`:
```python
def __init__(self, config: BaseModelProviderConfig):
    self.config = config
```

This means `user_id` is completely dropped from the provider hierarchy. However:

1. **`OpenAiLlmProvider.__init__`** (`llm_providers/openAi.py` line 8-9) currently calls `super().__init__(config, user_id)` — this `super()` call will break immediately.
2. **`llm_factory.py`** line 57 currently passes `user_id` when constructing the provider: `provider = LlmProviderClass(config=llm_config, user_id=user_id)`.
3. The spec redesigns the factory to only pass `bot_id` and `feature_name` to `create_model_provider`, and the factory internally resolves `user_id` via the resolver. But **where does `user_id` go** after that? It's needed for `TokenTrackingCallback` but no longer stored on the provider.

The spec must clarify: does `user_id` get resolved by the factory and used **only** for the callback construction, or is it still passed into the provider? The current spec snippet for the factory shows it being `await`ed from the resolver but never passed to the provider constructor — which is correct for the `BaseModelProvider` spec, but must be explicitly called out as an intentional change.

**Required Action**: 
- Add a note in Section 3.3 (`openAi.py`) that `OpenAiChatProvider.__init__` must be updated to remove the `user_id` parameter from both its own signature and the `super().__init__()` call. 
- Document that `user_id` is now resolved internally by the factory and used exclusively for the `TokenTrackingCallback`.

### <a id="gap-004"></a>GAP-004: Factory return type `Union[BaseChatModel, BaseModelProvider]` forces unsafe casting on every caller

**Priority**: 🔴 P1  
**ID**: GAP-004  
**Status**: pending

**Description**:  
The spec (Section 4.2) defines the factory return type as:
```python
async def create_model_provider(
    bot_id: str,
    feature_name: str,
    config_tier: Literal["high", "low", "image_moderation"]
) -> Union[BaseChatModel, BaseModelProvider]
```

The spec correctly notes that "any caller expecting a specific type *must* perform a runtime type check". However, this introduces several concerns:

1. **Every existing caller** (`AutomaticBotReplyService._initialize_llm`, `extractor.py`, `runner.py`) currently expects `BaseChatModel` and chains it into Langchain pipelines (`prompt | llm | StrOutputParser()`). None of them currently perform runtime checks. The spec should document that **existing callers requesting "high" or "low" tiers will always get `BaseChatModel`** — clarifying that the `Union` is really only polymorphic when `config_tier="image_moderation"` is used.

2. **Type ergonomics**: Mypy and Pyright will flag every usage of the returned value without a type-narrowing guard. Since "high" and "low" callers will **never** get a `BaseModelProvider`, consider using `@overload` decorators to provide tier-specific return types:
   ```python
   @overload
   async def create_model_provider(bot_id: str, feature_name: str, config_tier: Literal["high", "low"]) -> BaseChatModel: ...
   @overload
   async def create_model_provider(bot_id: str, feature_name: str, config_tier: Literal["image_moderation"]) -> BaseModelProvider: ...
   ```

3. **No example caller code** is shown for the `image_moderation` case. Who calls `create_model_provider(bot_id, ..., "image_moderation")` and how do they use the `BaseModelProvider`? The media processing pipeline (`media_processing_service.py`) currently has no integration point for moderation.

**Required Action**: 
- add a note that callers of "high"/"low" tiers can safely assume `BaseChatModel` return type. 
- Provide an example of how the `image_moderation` tier is consumed.

### <a id="gap-006"></a>GAP-006: `resolver.py` has no existence path — how does it get `db` / `configurations_collection`?

**Priority**: 🟠 P2  
**ID**: GAP-006  
**Status**: pending

**Description**:  
The spec (Section 4.3) introduces `services/resolver.py` with two functions:
- `async def resolve_user(bot_id: str) -> str`
- `async def resolve_model_config(bot_id: str, ...) -> Union[...]`

These functions need to query the database to look up bot configurations and owner users. However, the spec does not specify **how these functions obtain their database connection**:

1. Do they accept `db` as a parameter? This contradicts the factory's philosophy of reducing parameter passing.
2. Do they use `GlobalStateManager.get_instance()` internally? This would be consistent with the factory's approach (Section 4.2 mentions using `get_global_state().token_consumption_collection`), but is not stated.
3. Are they methods on a class, or standalone functions?

Additionally, the **resolution logic itself** is non-trivial:
- `resolve_user(bot_id)` must query `credentials_collection` to find which user has `bot_id` in their `owned_bots` array. This is the same pattern used in `QuotaService._stop_user_bots`.
- `resolve_model_config(bot_id, tier)` must query `configurations_collection` (or `bot_config_collection`) to fetch the bot's `config_data.configurations.llm_configs.<tier>` (or the renamed path).

The spec should provide a concrete implementation sketch or at least specify the data access pattern.

**Required Action**: 
- Clarify that resolvers use `GlobalStateManager.get_instance()` for database access (consistent with the factory). 
- research and provide the spec with query patterns (e.g., `credentials_collection.find_one({"owned_bots": bot_id})` for user resolution).
- research and provide the spec with querry patterns for model config resolution (from bot_configurations_collection)

### <a id="gap-007"></a>GAP-007: Inconsistent rename of `record_llm_interactions` across config hierarchy

**Priority**: 🟠 P2  
**ID**: GAP-007

**Description**:  
This is a sister issue to GAP-001 but focuses on the config model hierarchy design:

- `BaseModelProviderSettings` — api_key_source, api_key, model

Furthermore, the `LLMRecorder` class (imported in `extractor.py` line 85) also uses "LLM" terminology. Is `LLMRecorder` also being renamed?

**Required Action**: 
- same conclusion as GAP-001, this is too risky to change and has zero functional implications
- keep the field as `record_llm_interactions` since it only lives in `ChatCompletionProviderSettings` (which is exclusively for LLM chat completions).

### <a id="gap-009"></a>GAP-009: `AutomaticBotReplyService` async refactor — `SessionManager._initialize_components` is synchronous

**Priority**: 🟠 P2  
**ID**: GAP-009  
**Status**: pending

**Description**:  
The spec (Section 4.5.1) correctly identifies that `AutomaticBotReplyService.__init__` calls `create_tracked_llm` synchronously (line 189: `self._initialize_llm()`), and that this must be refactored since the factory is now async.

However, upon investigation, `AutomaticBotReplyService` is NOT created from a synchronous constructor chain. It is created inside `BotLifecycleService.create_bot_session()` (line 218), which is **already an async method**:
```python
async def create_bot_session(self, config: BotConfiguration) -> SessionManager:
    ...
    if config.features.automatic_bot_reply.enabled:
        bot_service = AutomaticBotReplyService(instance)  # line 218
        instance.register_message_handler(bot_service.handle_message)
```

Therefore, the spec's suggestion to introduce a separate `async def start()` method managed by an external service manager is **overkill**. The fix is simpler:

**Required Action**: 
- Remove the `self._initialize_llm()` call from `AutomaticBotReplyService.__init__` (line 189).
- Make `_initialize_llm` async: `async def _initialize_llm(self)`.
- Call `await bot_service._initialize_llm()` right after construction, inside the already-async `create_bot_session()`:
  ```python
  if config.features.automatic_bot_reply.enabled:
      bot_service = AutomaticBotReplyService(instance)
      await bot_service._initialize_llm()  # <-- simple addition
      instance.register_message_handler(bot_service.handle_message)
  ```
- Update the spec (Section 4.5.1) to reflect this simpler integration point instead of the generic "service manager" language.

### <a id="gap-010"></a>GAP-010: `BaseModelProvider` has no abstract methods — no contract enforcement for `moderate_image()`

**Priority**: 🟠 P2  
**ID**: GAP-010  
**Status**: pending

**Description**:  
The spec (Section 3.1) states that `BaseModelProvider` "defines **no abstract methods** on its own — subclasses are free to expose whichever interface is appropriate."

While this provides flexibility, it means `OpenAiModerationProvider.moderate_image()` has **no contractual enforcement**. Any code that receives a `BaseModelProvider` instance cannot safely call `moderate_image()` without runtime `hasattr()` or `isinstance()` checks.

Contrast this with `ChatCompletionProvider` which properly defines `get_llm()` as abstract — guaranteeing the method exists on all subclasses.

For consistency and type safety, consider introducing an intermediate `ImageModerationProvider(BaseModelProvider)` abstract class (parallel to `ChatCompletionProvider`) that defines `moderate_image()` as abstract:

```python
class ImageModerationProvider(BaseModelProvider):
    @abstractmethod
    async def moderate_image(self, image_url: str) -> ModerationResult:
        pass
```

This would also make the factory's return type cleaner: `Union[BaseChatModel, ImageModerationProvider]`.

**Required Action**: 
- Introduce `ImageModerationProvider` as an intermediate abstract class, mirroring the `ChatCompletionProvider` pattern. 
- This provides compile-time contract enforcement and cleaner typing.
